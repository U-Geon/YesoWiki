# YesoWiki 요구사항 정의서 (Requirements)

본 문서는 YesoWiki 프로젝트가 달성해야 할 기능적/비기능적 요구사항을 정의합니다.

## 1. 개요 (Overview)

- **목적**: 친구들끼리 서로의 정보, 밈(Meme), 에피소드 등을 자유롭게 등록하고 열람할 수 있는 폐쇄형 위키 서비스 구축.
- **주요 타겟**: 소규모 지인 그룹.

## 2. 기능 요구사항 (Functional Requirements)

### 2.1. 위키 문서 (Wiki Documents)

- **CRUD**: 새로운 문서를 생성, 읽기, 수정할 수 있어야 합니다. (삭제는 관리자 또는 작성자만 가능하거나, 히스토리만 남기고 숨김 처리)
- **마크다운 지원**: 문서 내용은 마크다운(Markdown) 포맷으로 작성되어야 하며, 이를 안전하게 렌더링해야 합니다.
- **백링크(Backlink)**: 문서 내에서 `[[다른 문서 제목]]` 형태로 작성 시 자동으로 해당 문서로 연결되는 하이퍼링크가 생성되어야 합니다.

### 2.2. 문서 히스토리 (Version History)

- **변경 이력 추적**: 문서가 수정될 때마다 기존 버전을 `DocumentHistory`에 백업하여 누가 언제 어떤 내용을 수정했는지 추적할 수 있어야 합니다.
- **(선택/확장) 리비전 비교**: 과거 버전과 현재 버전 간의 차이(Diff)를 볼 수 있는 기능.

### 2.3. 편집 정책 (Editing Policy)

- **익명 편집**: 로그인 기능 없이 접속한 **누구나** 문서를 열람하고 생성/수정할 수 있어야 합니다. (실제 나무위키와 동일한 방식)
- **기여자 식별**: 편집 시 편집자의 **IP 주소**를 자동 기록하고, 선택적으로 **닉네임**을 입력할 수 있어야 합니다.
- **히스토리 추적**: 누가(IP/닉네임) 언제 수정했는지 `DocumentHistory`에 기록하여 반달리즘(훼손) 시 복구 가능해야 합니다.

### 2.4. UI / UX

- **디자인 원칙**: 단순한 텍스트 나열이 아닌, 다크모드를 지원하고 타이포그래피와 여백이 강조된 프리미엄 모던 디자인을 적용해야 합니다.
- **반응형 웹**: 모바일 환경에서도 문서 열람 및 수정이 원활해야 합니다.

## 3. 비기능 요구사항 (Non-Functional Requirements)

- **성능 (Performance)**: Next.js의 Server Components와 Fetch 캐싱을 적극 활용하여 초기 로딩 속도와 SEO(내부망이라도 구조적 이점)를 최적화합니다.
- **확장성 (Scalability)**: Prisma ORM을 사용하여 향후 데이터 구조 변경 및 DB 마이그레이션이 용이해야 합니다.
- **배포 (Deployment)**: Vercel 환경에서 무중단으로 CI/CD가 이루어져야 하며, Vercel Edge Cache 등과 호환되어야 합니다.
- **데이터베이스**: 배포 환경에서는 **Supabase(PostgreSQL)**를 사용합니다. (`prisma/schema.prisma` provider 참조)
